# Endorsement Questionnaire System — Build Spec & Protocols

A hosted, autosaving version of the questionnaire editor that extends your existing
GitHub Pages + Supabase setup. Designed so **Claude Code does the generation** (research,
fact-check, suggested copy) and the **browser page just renders and autosaves**.

---

## The core idea

Your v3 editor is already the front end. Turning it into the hosted system is three swaps,
not a rebuild:

1. **Storage** — browser `window.storage` becomes Supabase. Autosave survives a Safari crash
   and syncs across devices.
2. **Content** — each org's questions, levers, sources, and suggested copy move from inline
   JSON to one file per org in the repo (`/questionnaires/{slug}.json`), generated by Claude
   Code when you ingest a new questionnaire.
3. **Reuse** — answers you write in one form surface as reference chips in others, because the
   page queries Supabase for your same-topic answers across all orgs. Live, refreshes on save
   and on an interval.

Everything else already exists in v3: the four panels, the "use this" buttons, the export.

---

## Architecture (what lives where)

| Piece | Where | Who writes it |
|---|---|---|
| The page (`questionnaire.html?org=nuhw`) | GitHub Pages (static) | Once, then rarely |
| Per-org content (questions, levers, sources, copy, research notes, references) | Repo `/questionnaires/{slug}.json` | Claude Code, on ingest |
| Your answers (drafts) | Supabase `answers` | The page, on every keystroke |
| Cross-org reuse | Supabase query over `answers` by topic | The page, live |
| Status for the tracker | Supabase `questionnaire_status` | The page + a sheet sync |

**Why the split:** answers change constantly, so they belong in a database. The generated
research is a point-in-time artifact you want versioned and diff-able, so it belongs in git.
This split also makes "no Claude anywhere" clean — the JSON is just data with neutral labels
(`analysis`, `research_notes`, `strategy`, `levers`, `sources`, `suggested`).

**No API in the browser.** All generation happens in Claude Code when you run a protocol.
The page never calls a model.

## Supabase — tables to ADD

Run this in the Supabase SQL editor. It only adds; it touches nothing you already have.

```sql
-- Your editable answers, autosaved from the page
create table if not exists questionnaire_answers (
  id           bigint generated always as identity primary key,
  org_slug     text not null,          -- 'nuhw', 'sierra_club', 'alc', ...
  question_id  text not null,          -- matches the id in the org JSON
  topic        text,                   -- 'immigration','housing','single_payer', ... for reuse
  answer_text  text default '',
  updated_at   timestamptz not null default now(),
  unique (org_slug, question_id)
);

-- Per-org status, for the tracker hook
create table if not exists questionnaire_status (
  org_slug      text primary key,
  display_name  text,
  status        text default 'not_started',  -- not_started | in_progress | submitted
  last_factcheck date,
  submitted_at  date,
  notes         text,
  updated_at    timestamptz not null default now()
);

-- keep updated_at fresh
create or replace function touch_updated_at() returns trigger as $$
begin new.updated_at = now(); return new; end; $$ language plpgsql;

drop trigger if exists t_answers on questionnaire_answers;
create trigger t_answers before update on questionnaire_answers
  for each row execute function touch_updated_at();
```

**Auth / RLS — the honest tradeoff.** Single-user tool, so the simplest path is: leave RLS
off (or add a permissive policy) and use the public anon key in the page. That means anyone
who has both the page URL and the anon key could read/write your drafts. For an internal
working tool that is usually fine. If you want a gate, the light option is a shared passphrase
that unlocks a Supabase Auth anonymous session before the page will read/write — one extra
screen, no per-user accounts. Recommend starting open, adding the passphrase only if the URL
ever gets shared.

## Per-org JSON contract

Claude Code writes one of these per questionnaire into `/questionnaires/{slug}.json`. The page
renders it. Shape (trimmed to essentials — mirrors your v3 panels):

```json
{
  "slug": "nuhw",
  "org": "National Union of Healthcare Workers",
  "cycle": "2026",
  "candidate": "Daria Wrubel — Berkeley City Council D1",
  "generated": "2026-07-22",
  "last_factcheck": "2026-07-22",
  "org_research": "What this org weighs, its mission, its current fights, its endorsement posture. 3-6 sentences.",
  "contact_fields": [
    { "k": "name", "label": "Name", "v": "Daria Wrubel" }
  ],
  "questions": [
    {
      "id": "s5a",
      "section": "Section 5A",
      "topic": "healthcare_standards",
      "status": "done",
      "question": "How would you advocate for the improvement of healthcare system standards?",
      "analysis": "Neutral read: what's strong, what's missing. No 'Claude' anywhere.",
      "references": [
        { "who": "Cecilia — SEIU 1021", "text": "How she handled the analogous question." }
      ],
      "levers": [
        { "tier": "authority", "text": "Real council power, verified." },
        { "tier": "advocacy",  "text": "Resolution / lobbying only." }
      ],
      "sources": [
        { "label": "City of Berkeley: BHSA", "url": "https://berkeleyca.gov/..." }
      ],
      "suggested": [
        { "note": "Drawn from her own writing + verified facts", "text": "Copy with [DARIA: ...] brackets where firsthand detail is needed." }
      ]
    }
  ]
}
```

Notes:
- `topic` is the join key for cross-org reuse. Keep a controlled list (housing, immigration,
  single_payer, behavioral_health, labor_law, etc.) so same-topic answers match across forms.
- `status` drives the dots and the progress bar, same as v3.
- Nothing here references a model. Labels are neutral.

## Frontend page — how it changes from v3

Take the v3 editor and make four edits:

1. **Load content by org.** Read `?org=` from the URL, `fetch('/questionnaires/'+org+'.json')`
   instead of the inline `<script id="qdata">`.
2. **Load answers from Supabase.** On load, select from `questionnaire_answers` where
   `org_slug = org`, merge into the editor.
3. **Autosave to Supabase.** Replace the `window.storage.set` debounce with an upsert into
   `questionnaire_answers`. Keep the same 350ms debounce.
4. **Reuse chips.** For the current question's `topic`, select answers from
   `questionnaire_answers` where `topic = current AND org_slug <> org AND answer_text <> ''`,
   render them as chips in a new "From your other questionnaires" panel. Refresh on save and
   every 5 minutes.

Supabase from a static page uses the JS client via CDN:

```html
<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>
<script>
  const sb = supabase.createClient(SUPABASE_URL, SUPABASE_ANON_KEY);
  // upsert on edit:
  await sb.from('questionnaire_answers')
    .upsert({ org_slug, question_id, topic, answer_text }, { onConflict: 'org_slug,question_id' });
  // load:
  const { data } = await sb.from('questionnaire_answers').select('*').eq('org_slug', org_slug);
  // reuse:
  const { data: reuse } = await sb.from('questionnaire_answers')
    .select('org_slug,answer_text')
    .eq('topic', topic).neq('org_slug', org_slug).neq('answer_text', '');
</script>
```

That's the whole app. The panels, buttons, dots, and export are unchanged.

## INGEST PROTOCOL — run this in Claude Code

Paste this when you drop a new questionnaire. Claude Code has repo access, web search, and can
read the source file, so it does the whole thing and commits.

```
INGEST a new endorsement questionnaire.

Inputs I will give you: the org name, and the source questionnaire (a PDF path, a Drive link,
or pasted text). If I only name the org, ask me where the source is.

Do all of this, then write /questionnaires/{slug}.json and commit:

1. PARSE. Extract every question verbatim, with its section structure and any yes/no or
   checklist items. Do not paraphrase questions.
2. ORG RESEARCH. Search the org's mission, current political priorities, active campaigns,
   and endorsement posture. Write a 3-6 sentence org_research note: what they weigh, what
   they want to hear, where the candidate already overlaps. Cite what you find.
3. REFERENCES. For each question, find the closest analogue in the reference set (Cecilia
   Lunaparra and Syrak Micael questionnaires — SEIU 1021, EBDSA, Sierra Club, etc.) and
   summarize how they handled it. These are framing references only; never import their
   biographical or record claims as Daria's.
4. FACT-CHECK + LEVERS. For each policy question, identify what a Berkeley city councilmember
   can actually do. Tier every lever: "authority" (real council power) vs "advocacy"
   (resolution or lobbying only). Verify each fact against a real, current source and include
   the link. Do not overstate authority. Prefer primary sources (city agendas, ordinances,
   ballotpedia, agency pages). Note honest caveats (e.g. a program was defunded, a count is
   contested).
5. REUSE. Read my existing answers from Supabase (questionnaire_answers) and from other
   /questionnaires/*.json. For each question, surface my own prior answers on the same topic
   so I can reuse language. Tag each question with a controlled `topic` so cross-form reuse
   works.
6. SUGGESTED COPY. For blank or thin questions, draft copy from my and Daria's actual writing
   plus the verified facts. Honest, no overselling, no em dashes. Where an answer needs Daria's
   firsthand clinical detail, insert a [DARIA: ...] bracket rather than inventing an anecdote.
   Label the footing of each draft (e.g. "her sentences, reordered" / "built from verified
   levers").
7. NEUTRAL LABELS. The output JSON must not reference Claude, AI, or any model anywhere. Use
   analysis / research_notes / strategy / levers / sources / suggested.
8. WRITE + COMMIT. Write /questionnaires/{slug}.json per the contract in
   questionnaire_system_spec.md. Upsert a row into questionnaire_status (org_slug,
   display_name, status='not_started', last_factcheck=today). Commit with a clear message.
9. TRACKER. Tell me the one line to paste into the Daria Wrubel Endorsement Tracker Orgs tab:
   the questionnaire URL and today's fact-check date.
```

## REFRESH PROTOCOL — run this to re-verify

Facts drift (dates, dollar figures, who holds a seat). Run this per org when a questionnaire
has been sitting a while, or before you submit.

```
REFRESH the fact-check for /questionnaires/{slug}.json.

1. For every source link in the file, re-verify the claim it supports is still current. Flag
   anything stale (a figure that changed, a program that closed, a bill that passed or failed).
2. Re-run the levers check for any question whose facts changed. Keep the authority/advocacy
   tiering honest.
3. Do NOT touch my answers in Supabase. Only update the generated content (analysis, levers,
   sources, suggested copy).
4. Update last_factcheck to today in both the JSON and questionnaire_status. Commit.
5. Give me a short changelog of what moved and why.
```

Reuse chips are live from Supabase, so they need no refresh. This protocol is only for the
generated research layer.

## Tracker hook

Add three columns to the **Orgs** tab of the Daria Wrubel Endorsement Tracker:
- **Questionnaire** — link to `…/questionnaire.html?org={slug}`
- **Q Status** — not started / in progress / submitted
- **Last fact-check** — date

Two ways to keep Q Status current:
- **Simple (recommended to start):** you update it by hand; the ingest protocol hands you the
  row to paste.
- **Automated (later):** a small Google Apps Script on a time trigger reads
  `questionnaire_status` from Supabase and writes the columns. Worth it only once you have
  several forms in flight.

## Build order

1. Run the Supabase SQL (2 tables). ~5 min.
2. Fork the v3 editor into `questionnaire.html` and make the four frontend edits. Test with the
   existing NUHW content pasted into `/questionnaires/nuhw.json`. This proves storage + reuse
   before any ingest.
3. Run the INGEST PROTOCOL on the Sierra Club questionnaire as the second form. This proves the
   generation pipeline and the cross-form reuse (NUHW ↔ Sierra Club share housing, immigration,
   labor topics).
4. Add the tracker columns.
5. Everything after that is: drop a questionnaire, run ingest, fill it in.
